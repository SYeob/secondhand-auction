# Pa-Bi Auction QA 프로젝트 개요

## 1. 프로젝트 목적

Pa-Bi Auction을 대상으로 요구사항 분석, 위험 기반 테스트 설계, UI/API 자동화, 결함 관리, CI 및 결과 보고까지 수행하는 경력직 QA 포트폴리오 프로젝트입니다.

자동화 코드의 양보다 다음 QA 사고 과정을 우선시 합니다.

- 금전·개인정보 위험을 기준으로 P0 범위 선정
- UI와 API 계층의 검증 책임 분리
- 역할별 계정과 독립적인 테스트 데이터 관리
- 실패 증적과 잔여 위험 기록
- 반복 실행 가능한 회귀 테스트 구성

## 2. 대상 서비스와 기술 구성

| 구분       | 구성                                         |
| ---------- | -------------------------------------------- |
| Frontend   | React, Vite, TypeScript                      |
| Backend    | Supabase Auth, PostgREST, PostgreSQL RLS/RPC |
| UI 자동화  | Python 3.11, Selenium 4.46, pytest           |
| API 자동화 | requests, pytest                             |
| 설정 관리  | python-dotenv, GitHub Actions Secrets        |
| 결과 보고  | pytest-html, 실패 스크린샷, Actions Artifact |
| CI         | GitHub Actions, Chrome Headless              |

## 3. 기능 우선순위

| 우선순위 | 기능                                         | 선정 이유                                      |
| -------- | -------------------------------------------- | ---------------------------------------------- |
| P0       | 입찰, 최고가, 경매 종료, 낙찰자, 연락처 권한 | 금전 분쟁과 개인정보 유출로 직접 연결          |
| P1       | 로그인, 상품 등록·조회, 마이페이지           | 핵심 흐름의 선행 조건과 사용자 데이터 확인     |
| P2       | 좋아요                                       | 거래 결과에는 영향이 적지만 반복 사용성에 영향 |

## 4. 핵심 품질 목표

1. 유효한 최고 입찰 금액과 사용자가 정확히 반영되어야 한다.
2. UI를 우회해도 현재가 이하 및 종료 후 입찰이 서버에서 차단되어야 한다.
3. 종료 시점의 최고 입찰자가 낙찰자가 되어야 한다.
4. 판매자 연락처는 낙찰자에게만 제공되어야 한다.
5. 테스트는 실행마다 고유 데이터를 생성하고 종료 후 정리해야 한다.

## 5. 자동화 구현 현황

| 구분 | 구현 테스트 수 | 주요 범위                                                |
| ---- | -------------: | -------------------------------------------------------- |
| UI   |             13 | Smoke, 로그인, 등록, 입찰, 종료·낙찰, 마이페이지, 좋아요 |
| API  |             11 | 인증, 입찰 유효성·동시성, 종료, 낙찰, RPC·컬럼 권한      |
| 합계 |             24 | P0/P1/P2 핵심 회귀                                       |

## 6. 포함·제외 범위

### 포함

- 정상·예외 사용자 흐름
- 인증과 인가
- 최고가 및 종료 상태
- 순차·동시 API 입찰
- Supabase RLS/Trigger/RPC 검증
- CI 회귀 실행과 증적 보존

### 제외

- 실제 결제, SMS, 이메일 발송
- 운영 규모 부하·장시간 내구성 테스트
- Supabase 자체 서비스 가용성 검증
- 모바일 실기기 및 전체 브라우저 조합

## 7. 포트폴리오 완료 기준

- P0 자동화 전체 실행 및 결과 확인
- 실패와 알려진 결함을 구분하고 결함 ID와 연결
- 테스트 계정과 키가 저장소에 노출되지 않음
- 로컬과 GitHub Actions에서 동일 명령으로 실행 가능
- HTML 리포트, 스크린샷, 잔여 위험이 문서화됨

제품 출시 승인은 별도의 품질 게이트로 판단하며 Critical 미해결 결함이 0건이어야 함.
