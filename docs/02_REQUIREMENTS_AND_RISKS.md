# 요구사항 및 위험 분석

## 1. 위험 평가 기준

위험 점수는 `발생 가능성(1~5) × 영향도(1~5)`로 계산한다.

| 점수 | 등급 | 대응 |
|---:|---|---|
| 15~25 | Critical | P0, UI와 API 모두 검증 |
| 8~14 | High | P0/P1 회귀 포함 |
| 4~7 | Medium | 핵심 정상 흐름 후 검증 |
| 1~3 | Low | 탐색 또는 선택 자동화 |

## 3. 계층별 검증 책임

| 검증 항목 | UI | API/DB |
|---|---|---|
| 입력 안내와 사용자 흐름 | 주 검증 | 보조 |
| 현재가 이하 입찰 | 오류 메시지 | Trigger 차단 필수 |
| 종료 후 입찰 | 버튼 비활성화 | 서버 차단 필수 |
| 최고가·최고 입찰자 | 화면 가격 | DB 정렬·가격 검증 |
| 연락처 권한 | 낙찰자 화면 | RPC 인증·인가 검증 |
| 마이페이지·좋아요 | 목록 반영 | RLS 데이터 범위 |

## 4. 테스트 데이터 위험과 통제

- 판매자, 입찰자 A, 입찰자 B는 서로 다른 계정을 사용한다.
- 비밀번호와 Supabase 키는 `.env` 또는 Actions Secrets에서만 주입한다.
- 상품명에 UTC 마이크로초를 포함해 충돌을 방지한다.
- 각 테스트는 자신이 만든 상품만 삭제한다.
- 상품 삭제의 cascade 정책으로 관련 입찰·좋아요도 정리한다.

## 5. 확인된 요구사항 불일치

| 결함 ID | 내용 | 영향 | 상태 |
|---|---|---|---|
| DEF-001 | 화면은 이미지 URL을 선택사항으로 안내하지만 DB 스키마는 `image_url NOT NULL` | 이미지 URL 미입력 시 정상 등록 차단 가능 | Open |
| DEF-002 | 익명 REST 요청으로 `products.seller_phone` 컬럼 조회 가능 | 판매자 연락처 개인정보 유출 | Open / Critical |

자동화 정상 흐름에서는 유효한 테스트 이미지 URL을 입력해 다른 핵심 검증을 계속한다. 별도 결함 재현과 제품 수정 후 회귀가 필요하다.

`DEF-002`는 공개 배포 환경에 읽기 전용 요청을 보내 HTTP 200 응답을 확인했다. 애플리케이션이 해당 컬럼을 선택하지 않더라도 REST API 직접 호출로 우회할 수 있으므로 출시 차단 수준으로 판단한다.

## 6. 잔여 위험

- 분산 환경에서 서버 간 시각 차이
- 실제 대규모 동시 입찰의 잠금·지연 특성
- 중복 클릭 및 네트워크 재전송에 대한 멱등성
- 장기간 누적 데이터에서의 조회 성능
- 모바일 브라우저 및 접근성 품질
