# 핵심 테스트 케이스

## 1. TC-AUTH-001 정상·실패 로그인

| 항목 | 내용 |
|---|---|
| 사전 조건 | 등록된 테스트 계정, 실행 중인 Vite 서버 |
| 입력 | 정상 비밀번호 / 정상 비밀번호에 `_invalid`를 붙인 값 |
| 기대 결과 | 정상 계정은 로그아웃 버튼 표시, 잘못된 비밀번호는 인증 오류 |
| 자동화 | UI 및 Auth API |

## 2. TC-ITEM-001 정상 상품 등록

| 항목 | 내용 |
|---|---|
| 사전 조건 | 로그인 사용자 |
| 절차 | 상품명, 위치, 시작가, 이미지 URL, 미래 종료 시간 입력 |
| 기대 결과 | 홈 목록에 고유 상품명 표시 후 본인 상품 삭제 가능 |
| 정리 | `finally`에서 생성 상품 삭제 |
| 자동화 | UI |

| 항목 | 내용 |

| 사전 조건 | 시작가 10,000원, 판매자와 다른 입찰자 |

| 기대 결과 | 입찰 저장, 현재가 15,000원, 완료 메시지 표시 |
| 자동화 | UI 및 API |

## 4. TC-BID-002 현재가 이하 입찰

| 항목 | 내용 |
|---|---|
| 입력 | 10,000원 / 9,000원 |
| 기대 결과 | UI 오류, API 거부, 입찰 내역 미생성, 현재가 10,000원 유지 |
| 자동화 | UI parametrization 및 API parametrization |

## 5. TC-BID-003 복수·동시 입찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 입찰자 A와 B, 동일 진행 경매 |
| 입력 | A 15,000원, B 20,000원 |
| 절차 | 순차 요청 및 ThreadPoolExecutor 동시 요청 |
| 기대 결과 | 처리 순서와 무관하게 현재가 20,000원, 최고 입찰자 B |
| 자동화 | API |

## 8. TC-PRIV-001 판매자 연락처 권한

| 항목 | 내용 |
|---|---|
| 사전 조건 | 연락처가 있는 종료 경매와 낙찰자 B |
| 절차 | B, 비낙찰자 A, 익명 요청으로 `get_seller_contact` 호출 |

## 9. TC-PRIV-002 연락처 컬럼 직접 접근

| 항목 | 내용 |
|---|---|
| 사전 조건 | 연락처가 저장된 상품 |
| 절차 | 익명 API 키로 `products?select=seller_phone` 직접 호출 |
| 기대 결과 | 401 또는 403으로 컬럼 접근 거부 |
| 실제 결과 | HTTP 200, DEF-002 |
| 자동화 | strict xfail: 수정 후 XPASS를 실패로 처리 |

## 12. 공통 판정 규칙

- HTTP 성공 여부만으로 Pass 처리하지 않고 응답 데이터와 DB 조회 결과를 함께 확인한다.
- 개인정보 값은 허용 사용자 화면에서만 검증하고 로그에 출력하지 않는다.
- 생성 데이터 삭제 실패도 테스트 격리 실패로 취급한다.
- 서버 차단이 필요한 항목은 UI 테스트 통과만으로 완료 처리하지 않는다.
