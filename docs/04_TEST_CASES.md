# 핵심 테스트 케이스

## 1. 로그인

### TC-LOGIN-001 정상 로그인

| 항목 | 내용 |
|---|---|
| 사전 조건 | 등록된 사용자 계정이 존재한다. |
| 절차 | 로그인 화면에서 유효한 이메일과 비밀번호를 입력하고 로그인한다. |
| 기대 결과 | 로그인에 성공하고 인증 사용자 화면으로 이동한다. |
| 자동화 | 대상 |

### TC-LOGIN-002 보호 화면 접근 제어

| 항목 | 내용 |
|---|---|
| 사전 조건 | 로그아웃 상태이다. |
| 절차 | 마이페이지 또는 상품 등록 URL에 직접 접근한다. |
| 기대 결과 | 보호된 화면이 노출되지 않고 로그인 화면으로 이동한다. |
| 자동화 | 대상 |

## 2. 상품 등록

### TC-ITEM-001 정상 상품 등록

| 항목 | 내용 |
|---|---|
| 사전 조건 | 판매자 계정으로 로그인한다. |
| 절차 | 필수 정보, 시작가, 종료 시간을 입력하여 상품을 등록한다. |
| 기대 결과 | 등록에 성공하고 목록과 상세 화면에 동일한 정보가 표시된다. |
| 자동화 | 대상 |

### TC-ITEM-002 잘못된 종료 시간

| 항목 | 내용 |
|---|---|
| 사전 조건 | 상품 등록 화면에 접근한다. |
| 절차 | 현재 시각 이전의 종료 시간을 입력하고 등록한다. |
| 기대 결과 | 등록이 거부되고 종료 시간 오류가 표시된다. |
| 자동화 | 대상 |

---

## 3. 상품 조회

### TC-VIEW-001 목록과 상세 정보 일치

| 항목 | 내용 |
|---|---|
| 사전 조건 | 진행 중인 경매 상품이 존재한다. |
| 절차 | 목록에서 상품을 선택하고 상세 화면으로 이동한다. |
| 기대 결과 | 상품명, 현재 가격, 상태, 종료 시간이 목록과 상세 화면에서 일치한다. |
| 자동화 | 대상 |

### TC-VIEW-002 존재하지 않는 상품 접근

| 항목 | 내용 |
|---|---|
| 사전 조건 | 존재하지 않는 상품 ID를 준비한다. |
| 절차 | 해당 상세 URL에 직접 접근한다. |
| 기대 결과 | 사용자 친화적인 오류 또는 목록 이동 처리가 수행된다. |
| 자동화 | 선택 |

## 4. 입찰

### TC-BID-001 정상 입찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 로그인 상태이며 진행 중인 경매가 존재한다. |
| 절차 | 현재 최고가보다 높은 유효 금액을 입력하고 입찰한다. |
| 기대 결과 | 입찰에 성공하고 현재 가격과 최고 입찰자가 갱신된다. |
| 자동화 | 대상 |

### TC-BID-002 최고가 이하 입찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 현재 최고가가 설정되어 있다. |
| 절차 | 최고가와 같거나 낮은 금액으로 입찰한다. |
| 기대 결과 | 입찰이 거부되고 유효한 최소 입찰 금액이 안내된다. |
| 자동화 | 대상 |

### TC-BID-003 중복 입찰 요청

| 항목 | 내용 |
|---|---|
| 사전 조건 | 유효한 입찰 금액을 입력한 상태이다. |
| 절차 | 입찰 버튼을 빠르게 여러 번 클릭한다. |
| 기대 결과 | 하나의 입찰만 처리되며 중복 데이터가 생성되지 않는다. |
| 자동화 | API 중심 |

### TC-BID-004 동시 입찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 서로 다른 두 사용자와 동일 경매가 준비되어 있다. |
| 절차 | 두 사용자가 거의 동시에 서로 다른 금액으로 입찰한다. |
| 기대 결과 | 서버 처리 기준에 따라 더 높은 유효 입찰만 최고가로 반영된다. |
| 자동화 | API 또는 병렬 실행 |

### TC-BID-005 종료 이후 입찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 종료 시점이 지난 경매가 존재한다. |
| 절차 | UI와 API를 통해 입찰을 시도한다. |
| 기대 결과 | 입찰이 거부되고 기존 낙찰 결과가 변경되지 않는다. |
| 자동화 | 대상 |

## 5. 낙찰 및 경매 종료

### TC-CLOSE-001 정상 경매 종료

| 항목 | 내용 |
|---|---|
| 사전 조건 | 종료 예정 경매와 하나 이상의 입찰이 존재한다. |
| 절차 | 종료 시점이 지난 후 상태와 입찰 기능을 확인한다. |
| 기대 결과 | 경매 상태가 종료로 변경되고 추가 입찰이 차단된다. |
| 자동화 | 대상 |

### TC-CLOSE-002 최고 입찰자 낙찰

| 항목 | 내용 |
|---|---|
| 사전 조건 | 서로 다른 사용자의 입찰 내역이 존재한다. |
| 절차 | 경매 종료 후 낙찰자와 낙찰 금액을 확인한다. |
| 기대 결과 | 가장 높은 유효 금액을 입찰한 사용자가 낙찰자로 선정된다. |
| 자동화 | API 중심 |

### TC-CLOSE-003 유찰 처리

| 항목 | 내용 |
|---|---|
| 사전 조건 | 입찰 내역이 없는 경매가 존재한다. |
| 절차 | 종료 시간이 지난 후 경매 상태를 확인한다. |
| 기대 결과 | 낙찰자 없이 유찰 상태로 표시된다. |
| 자동화 | 대상 |

### TC-CLOSE-004 거래 정보 권한

| 항목 | 내용 |
|---|---|
| 사전 조건 | 낙찰 완료된 경매가 존재한다. |
| 절차 | 낙찰자, 판매자, 일반 사용자 계정으로 거래 정보를 조회한다. |
| 기대 결과 | 낙찰자와 판매자만 필요한 정보를 조회할 수 있다. |
| 자동화 | UI/API |

### TC-CLOSE-005 종료 경계 시점

| 항목 | 내용 |
|---|---|
| 사전 조건 | 종료 시간이 임박한 경매가 존재한다. |
| 절차 | 종료 직전과 직후에 각각 입찰을 요청한다. |
| 기대 결과 | 서버 종료 시각 기준으로 종료 이전 요청만 정책에 맞게 처리된다. |
| 자동화 | API 중심 |

| 자동화 | 대상 |

### TC-MY-002 다른 사용자 데이터 접근

| 항목 | 내용 |
|---|---|
| 사전 조건 | 서로 다른 두 사용자 계정이 존재한다. |
| 절차 | URL 또는 API 식별자를 변경해 다른 사용자의 데이터를 요청한다. |
